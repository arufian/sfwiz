# フローの命名規則とベストプラクティス

## デプロイ時のステータス

**フローは必ず非有効化状態（`<status>Draft</status>`）でデプロイすること。**

デプロイ時に `<status>Active</status>` を設定すると、デプロイ直後にフローが本番稼働状態になり予期しない自動化が実行されるリスクがある。有効化はデプロイ後にユーザーが手動で「有効化」ボタンを押して行う。

```xml
<!-- フローメタデータXML（*.flow-meta.xml）内の記述 -->
<status>Draft</status>  <!-- ← 常にDraftでデプロイ -->
```

## 基本形

### リソースにプレフィックスを付ける

フロー内のリソースには以下の命名規則でプレフィックスを付ける。

| 要素 | プレフィックス | API名の例 |
|---|---|---|
| 画面 | Screen | ScreenLeadUpdate |
| 画面の入力要素 | Input | InputFirstName |
| 画面の表示要素 | Display | DisplayErrorMessage |
| 数式 | Formula | FormulaToday |
| 変数 | Var（※recordIdは例外） | VarAccountId |
| 決定 | Decision | DecisionProfile |
| ループ | Loop | LoopContacts |
| 割り当て | Assign | AssignAccountShippingAddress |
| レコードを取得 | Select（オブジェクト名を含める） | SelectOpportunity |
| レコードを作成 | Insert（オブジェクト名を含める） | InsertLead |
| レコードを更新 | Update（オブジェクト名を含める） | UpdateContact |
| レコードを削除 | Delete（オブジェクト名を含める） | DeleteAccount |

決定内の分岐条件は `Is~`、`Has~` などの真偽値を表す名称を使用する。

### パスカルケースにする

すべてのリソース名はパスカルケース（各単語の先頭を大文字）にする。
例：`SelectOpportunity`、`DecisionIsManager`、`VarAccountId`

### 単純な分岐での変数割り当ては数式を使う

単純な条件分岐で異なる値を返すだけなら、決定要素を使わずに数式で表現する。

```
// ❌ Bad：単純な値の切り替えに決定要素を使う
[決定] → [割り当てA] / [割り当てB]

// ✅ Good：数式で条件に応じた値を返す
IF(条件, 値A, 値B) → 割り当て要素に直接設定
```

### 同じ処理をする分岐はまとめ、違う処理をする分岐だけ分ける

複数の分岐が「同じ処理（例：メッセージを表示する）」を行い、違うのは使うデータだけの場合、分岐ごとに画面やアクションを作るのではなく、**数式で表示内容を決めてから共通の画面・アクションに流す**。

分岐を作るのは「処理そのものが異なる」場合のみ。

**例：Apexの戻り値（1〜9）によってメッセージを切り替えるケース**

```
// ❌ Bad：戻り値ごとに別々の画面要素を作る（9つの画面が生まれる）
[決定] → (ret=1) → [Screen: MSG_008]
       → (ret=3) → [Screen: MSG_013]
       → (ret=4) → [Screen: MSG_014]
       → (ret=5) → [Screen: MSG_015]
       ... // 分岐数 = メッセージの種類数

// ✅ Good：メッセージを数式で決め、画面は1つにまとめる
[数式 FormulaMessage]
  = IF({!VarReturnCode} = 1, {!$Label.MSG_008},
    IF({!VarReturnCode} = 3, {!$Label.MSG_013},
    IF({!VarReturnCode} = 4, {!$Label.MSG_014},
    IF({!VarReturnCode} = 5, {!$Label.MSG_015}, ...))))

[決定 DecisionNeedsSpecialHandling]
  → (ret=2) → [特殊処理ブランチ：確認ダイアログ + OAuth誘導など]
  → (それ以外) → [Screen: {!FormulaMessage} を表示する共通画面]
```

**判断基準まとめ：**

| ケース | 対応方法 |
|---|---|
| 分岐ごとに表示する内容が違うだけ | 数式で内容を決め、画面・アクションは共通化 |
| 分岐ごとに処理（ナビゲーション・API呼び出し等）が違う | 分岐を残す |
| 2値の切り替えのみ | `IF(条件, 値A, 値B)` 数式で決定要素を不要にする |

## 高速更新 vs アクションと関連レコード

| | 高速項目更新 | アクションと関連レコード |
|---|---|---|
| 実行タイミング | レコードのDB保存処理の**前** | DB保存処理の**後** |
| 使用場面 | フローをトリガーしたレコードを更新する場合 | それ以外のすべての状況 |

**高速項目更新の注意点：**
- 「保存前更新」とも呼ばれ、レコード保存の前に実行される
- トリガーしたレコードを更新する場合に使用するが、実行順序によっては「アクションと関連レコード」を使用する場合もある

### 高速更新は機能単位に切り分ける

高速更新は厳密にはクエリを発行しないため、一つのフローに分岐を増やして可読性を下げるより、機能単位でフローを分けることを優先する。フロー数が増えても良い。

### トリガーレコードから辿れる情報は `$Record` を使用する

トリガーフローでは `$Record` グローバル変数でトリガー対象のオブジェクト情報を取得できる（`レコードを取得` 要素は不要）。

### 画面フローでは `recordId` 変数を使用する

`recordId` という変数を作成し、入力チェックをすることで、埋め込んだページのレコード情報を取得できる。

## ガバナ制限を意識する

### トランザクション単位のフロー制限

| 制限 | 値 |
|---|---|
| 発行されるSOQLクエリの合計数 | 100 |
| SOQLクエリで取得されるレコードの合計数 | 50,000 |
| 発行されるDMLステートメントの合計数 | 150 |
| DMLで処理されるレコードの合計数 | 10,000 |
| Salesforceサーバーの最大CPU時間 | 10,000ミリ秒 |
| 1バッチで許可される重複更新の合計数 | 12 |

**ループ内でデータ要素（レコード取得・作成・更新・削除）を使用しない。**
ループ外でコレクションに追加し、ループ後に一括処理する。

## テクニック系

### プラットフォームイベントでトランザクションを分ける

プラットフォームイベントを使用することで、重いトランザクションを非同期に分離できる。ガバナ制限に近づいた場合や、保存後の処理を非同期にしたい場合に有効。

### スケジュールトリガーの種類

- **バッチ起動** — 定期的に大量レコードをバッチ処理
- **バッチ起動（レコード単位）** — オブジェクトの各レコードを単位として処理
- **特定項目の時間差起動（スケジュール済みパス）** — 項目の値から指定時間後に実行

### エラーの通知先

デフォルトではフローのエラーメール通知は**フローの最終更新者**に届く。

任意の受信先を設定するには：
1. 「Apex例外メールの受信者」を設定する
2. プロセスの自動化設定でエラーメールの通知先を「Apex例外メールの受信者」に変更する

**注意：** この設定を変更するとすべてのフローのエラー通知が影響を受けるため、慎重に設定すること。
